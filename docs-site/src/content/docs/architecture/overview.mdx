FoundryGate is a governance and access-management layer that sits in front of your existing Azure AI Foundry deployment. It does not replace APIM or Foundry — it adds per-developer budget enforcement, individual key lifecycle management, and Entra-driven provisioning on top of them.

## Entra ID sync — how access follows your org chart

FoundryGate uses the Microsoft Graph API (app-only credentials stored in Key Vault) to keep developer access in sync with your Entra directory. This is how access governance works without any manual admin work after initial setup.

### First login — automatic provisioning

When a developer signs in through the Blazor UI for the first time, `GET /users/me` triggers the full provision pipeline:

```
1. Extract Entra object ID (oid claim) from the bearer token
2. Look up developer in the FoundryGate database
3. If not found → call Graph GET /users/{oid}
     → fetch displayName, mail, employeeId
     → INSERT User row
     → resolve monthly token quota (5-level resolution below)
     → call APIM Management: POST /subscriptions
         name: foundrygate-{userId}
         scope: /products/foundrygate
     → store encrypted APIM key in SQL (Key Vault key-wrapping)
     → INSERT QuotaAllocation for current period
4. Return UserProfileDto with masked key and current quota
```

The developer never needs to contact an admin. Their key is ready the moment they first open the app.

### Entra group sync — bulk provisioning and deprovisioning

Admins can link a FoundryGate group to an Entra group ID. Running `POST /groups/{id}/sync-entra` (or triggering from the UI) calls:

```
Graph GET /groups/{entraGroupId}/members
  → For each member in Entra:
      if not in FoundryGate DB → INSERT User
                                 (no APIM key yet — key on first login)
      if already exists        → update display fields
  → For members no longer in the Entra group:
      APIM Management: DELETE /subscriptions/{sid}
      User.IsActive = false
      QuotaAllocation.IsHardStopped = true
```

<Aside type="caution">
**Departures are hard**: when a developer leaves the Entra group, their APIM subscription is *deleted* — not just suspended. Their key stops working immediately. This is intentional. Quota exhaustion only suspends; Entra departure permanently revokes.
</Aside>

### Bulk user sync

`POST /users/sync` pages through all users assigned to the FoundryGate service principal via Graph and upserts them. New users get a record with the default quota; departed users are deactivated. Useful for an initial import of your entire org before any developer has logged in.

## Five-level quota resolution

When a developer's allocation is created or updated, `IQuotaResolutionService` walks this chain — first match wins:

```
1. User.IsUnlimited = true               → unlimited (skip everything)
2. User.MonthlyTokenQuota is set         → use the user-specific value
3. Any group the user belongs to has
   IsUnlimited = true                    → unlimited
4. Max(Group.MonthlyTokenQuota) across
   all the user's groups                 → use the highest group quota
5. SystemConfiguration
   ["DefaultMonthlyTokenQuota"]          → system-wide default (e.g. 1M)
```

The resolved value is written to `QuotaAllocation` for the current period. Admins can see which level resolved each developer's quota in the user detail view.

## APIM enforcement — how limits actually block traffic

FoundryGate doesn't rely on a database flag being checked on every request. The enforcement is APIM-native:

```
UsageSyncFunction  (runs every 15 minutes)
  ├── Queries Log Analytics: token totals per APIM subscription ID
  │   for the current billing period
  ├── Updates QuotaAllocation.TokensUsed in SQL
  └── For each developer where TokensUsed ≥ AllocatedTokens:
        APIM Management: PATCH /subscriptions/{sid}?state=suspended
        QuotaAllocation.IsHardStopped = true

Result: APIM returns 401 immediately. No Foundry traffic passes.
```

There is up to a 15-minute window between hitting the limit and enforcement. This is intentional — querying Log Analytics on every request would add latency to every AI call.

```
MonthlyResetFunction  (00:01 UTC on the 1st of each month)
  ├── For each active developer:
  │     Resolve quota for new period → INSERT QuotaAllocation
  │     If subscription was suspended → PATCH → state: active
  │     IsHardStopped = false
  └── Write audit log: quota.monthly-reset
```

## APIM key lifecycle

```
                    ┌─────────────────┐
                    │   No key yet    │
                    └────────┬────────┘
                             │ first login  OR  admin provisions
                             ▼
                    ┌─────────────────┐
                    │     Active      │ ◄── monthly reset re-enables
                    └────┬───────┬───┘
                         │       │
        quota exhausted  │       │  admin deactivates / Entra departure
                         ▼       ▼
               ┌──────────┐  ┌──────────┐
               │ Suspended│  │ Deleted  │
               │ (quota)  │  │(inactive)│
               └──────────┘  └──────────┘
                  Resets      Re-activation runs full provision pipeline
                  on 1st      (new key, new allocation)
```

**Suspend** — quota exhaustion: APIM subscription state → `suspended`. Key exists. Automatically re-enabled at monthly reset.

**Delete** — admin deactivation or Entra departure: APIM subscription deleted entirely. Key stops working immediately and cannot be restored without a full re-provision.
